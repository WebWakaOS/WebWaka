### 4. TEMPLATE FOR EACH EXECUTION PROMPT BLOCK

For every task block (per pre‑vertical area or per vertical), structure the prompt as follows.

You will **copy‑paste this template into each block** and then fill in the concrete details for that specific task.

#### 4.1. Block header (required)

Start with a clear header:

```markdown
### TASK: [Short task name]

- **Module / vertical**: e.g., “pre‑verticals / wallets”, “Vertical / Commerce POS”
- **GitHub context**:  
  - Docs: https://github.com/webwaka/.../blob/main/...
  - Code: https://github.com/webwaka/.../...
  - RFCs: https://github.com/webwaka/rfcs/...
```

#### 4.2. Agent skills & persona (required)

Define exactly who this agent is and what skills it must have:

```markdown
You are an expert [e.g., “Senior Backend Engineer specializing in Laravel microservices and Postgres schemas”] working on WebWaka’s platform.

Skills required:
- Deep understanding of WebWaka architecture and pre‑verticals (auth, billing, data, observability).
- Proficiency in WebWaka’s stack (e.g., Laravel, PostgreSQL, Redis, API‑gateway, event‑driven patterns).
- Experience with AI‑infrastructure patterns (credit‑based usage, wallet services, LLM orchestration).
- Solid grasp of GitHub best practices (branches, PRs, CI/CD, changelogs).
```

Adjust the skill list per task (e.g., swap in “frontend”, “mobile”, “DevOps”, “Data Engineer”, “Automation QA”, etc.).

#### 4.3. Mandatory deep context reading (100% non‑negotiable)

Every prompt must contain a **strong, non‑negotiable instruction** that the agent reads everything first:

```markdown
1. **Mandatory context reading (100% before execution):**

You must read and fully internalize the following documents and code BEFORE starting any implementation:

- Documentation:  
  - https://github.com/webwaka/.../blob/main/...
  - https://github.com/webwaka/.../blob/main/...
- Architecture & RFCs:  
  - https://github.com/webwaka/rfcs/blob/main/rfc-...-...
- Codebase references:  
  - https://github.com/webwaka/core/...  
  - https://github.com/webwaka/ai/...  
  - https://github.com/webwaka/verticals/commerce/...

Your understanding must be **100% grounded in these documents and code**; no decisions may be based on assumptions. If information is missing, unclear, or contradictory, you must first flag this in the issue/ticket or in a dedicated clarification thread before proceeding.
```

(You can adjust the URLs per task.)

#### 4.4. Research and execution‑plan step

Instruct the agent to research, then plan:

```markdown
2. **Online research and execution plan:**

Once you have a solid understanding of the context above:

- Conduct targeted online research around:
  - Best practices for: [e.g., “multi‑tenant wallet services”, “credit‑based AI usage metering”, “LLM orchestration with aggregators”].
  - Similar patterns in platforms like OpenRouter, Together, Eden, or other AI aggregators.
  - Recommended security, rate‑limiting, and audit patterns for wallet‑like services.
- Synthesize your research with the WebWaka context and prepare a short **execution plan** in this format:

  - **Objective**: 1–2 sentences.
  - **Key steps** (numbered):
    1. ...
    2. ...
  - **Risks and open questions**:
    - ...

Then, execute that plan.
```

Emphasize that **thoroughness matters far more than speed**:
- You may take as much time as needed to read docs, review code, and research.
- **Do not rush** to implement without a well‑grounded plan.

#### 4.5. Implementation instructions

Define the execution workflow:

```markdown
3. **Implementation workflow:**

Following your execution plan:

- Create a new branch off main (or the appropriate base branch) with a clear name that reflects the task.
- Implement:
  - Any required schema changes (migrations, database models).
  - New API endpoints or changes to existing ones.
  - New or modified WebWaka SuperAgent integration points (e.g., routing, credit‑burn logging, wallet updates).
  - Any required configuration flags, environment variables, or feature toggles.
- Ensure all code:
  - Follows WebWaka’s coding standards and linting rules.
  - Contains clear inline comments for complex logic.
- Write or update:
  - API contracts (OpenAPI or similar) where applicable.
  - READMEs or onboarding docs for the new service/module.
```

Make sure every task block tailors this to its specific domain (e.g., frontend, infra, data, etc.).

#### 4.6. QA and verification instructions (must be explicit)

Define a **QA block** that another agent (or the same agent playing the QA role) will execute:

```markdown
4. **QA and verification:**

You must now act as a **Senior QA Engineer** with deep experience in:

- Functional and non‑functional testing for SaaS platforms.
- API testing, integration testing, and edge‑case design.
- Performance and reliability testing for AI‑intensive services.

Tasks:

- Design a detailed test plan for this feature/module, including:
  - Positive test cases (happy paths).
  - Negative test cases (edge cases, error states, boundary values).
  - Security‑related tests (unauthorized access, rate‑limit bypass attempts, credit‑burn anomalies).
  - Recovery and failure‑mode tests (e.g., wallet service outage, aggregator unavailability).
- Implement automated tests where appropriate (unit, integration, end‑to‑end, or contract tests).
- Execute tests and document:
  - Pass/fail status.
  - Any bugs or issues found.
  - Recommended fixes or follow‑up work.
- Verify:
  - That all WebWaka SuperAgent integration points work end‑to‑end.
  - That credits are burned correctly and logged in the wallet and usage‑event system.
  - That BYOK paths (where applicable) do not interfere with WebWaka‑managed credits.
```

You can also add a **QA‑agent skill block** separate from the implementation‑agent skills:

```markdown
QA‑agent skills (recommended for this block):

- Senior QA Engineer with 5+ years in SaaS and API‑driven platforms.
- Strong experience with:
  - API‑testing frameworks (e.g., Postman, Cypress, or similar).
  - Test automation and CI/CD integration.
  - Performance and load‑testing tools.
- Familiarity with WebWaka’s testing stack and conventions.
```

#### 4.7. GitHub push and cleanup

Every block must end with a **GitHub‑push instruction**:

```markdown

- Do NOT leave anything important locally that is not pushed.
- Create a PR with:
  - A clear title summarizing the change.
  - A detailed description that references:
    - The relevant RFCs and documents.
    - High‑level impact on WebWaka SuperAgent and the vertical.
  - A checklist of items covered (e.g., implementation, tests, documentation).
- Request review from the appropriate WebWaka team members (as per the repo’s OWNERS or CODEOWNERS).
- Once approved and merged, mark the task as complete in the tracking system.
```

Emphasize that **no local partial work should be left behind**; everything must be in the repo.
