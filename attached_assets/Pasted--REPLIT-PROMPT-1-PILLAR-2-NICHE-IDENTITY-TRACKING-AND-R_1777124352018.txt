PHASE 3 — DEFINE THE NICHE IDENTITY SYSTEM
==================================================

Now design a canonical niche identity system for Pillar 2 template execution.

Purpose:
Future AI agents must be able to identify exactly one niche target at a time and know:
- the parent vertical,
- the niche name,
- its canonical code,
- whether template work has started,
- whether the niche is blocked,
- whether it is complete,
- whether variants exist,
- what should be worked on next.

You must define:
A. terminology model
- what is a vertical
- what is a niche
- what is a template family
- what is a template variant
- what is a runtime template vs marketplace template artifact
- what is a niche completion unit

B. identity model
Each niche must get a unique stable ID with a durable naming convention.

Design a robust format such as:
P2-{VERTICALCODE}-{NICHECODE}
or another better format if justified.

Your system must support:
- stable sorting,
- machine readability,
- human readability,
- future addition of more niches,
- future addition of more template variants,
- future “done / pending / blocked / in-review / approved / shipped” statuses,
- and zero ambiguity across the repo.

C. required fields for each niche record
At minimum, every niche record should include:
- nicheId
- verticalSlug
- verticalName
- nicheSlug
- nicheName
- canonicalPillars
- pillar2Eligible (boolean)
- nigeriaFirstPriority (high/medium/low or similar)
- africaFirstNotes
- audienceSummary
- businessContextSummary
- contentLocalizationNotes
- imageArtDirectionNotes
- regulatoryOrTrustNotes
- templateStatus
- templateVariantCount
- primaryTemplatePath
- marketplaceManifestSlug if applicable
- runtimeIntegrationStatus
- researchStatus
- lastReviewedAt
- dependencies
- blockers
- nextAction
- owner
- notes

D. status model
Define exact allowed statuses, for example:
- UNASSESSED
- READY_FOR_RESEARCH
- RESEARCH_IN_PROGRESS
- RESEARCH_SYNTHESIZED
- READY_FOR_IMPLEMENTATION
- IMPLEMENTATION_IN_PROGRESS
- IMPLEMENTED
- VERIFIED
- BLOCKED
- NEEDS_REVISION
- VARIANTS_PENDING

Refine this if needed, but make it strict.
