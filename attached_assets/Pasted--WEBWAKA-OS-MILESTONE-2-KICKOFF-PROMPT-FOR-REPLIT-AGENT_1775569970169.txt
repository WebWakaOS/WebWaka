## REQUIRED IMPLEMENTATION ORDER

Follow this order unless a blocker makes a small reorder necessary:

### Phase A — Verify and scaffold the monorepo runtime
Implement or complete:
- workspace/package manager setup
- TypeScript base config

- package naming conventions

Expected result:

---

### Phase B — Implement `packages/types`
Create canonical TypeScript interfaces/types for the approved root entities and key shared concepts.

At minimum include:
- Individual
- Organization
- Place
- Offering
- Profile
- Workspace
- BrandSurface

Also include shared supporting types for:
- Entity IDs
- Entity type enums
- Claim lifecycle states
- Subscription/entitlement enums or interfaces
- Geography type enums
- Political office type enums
- Territory type enums
- Verification states
- Publication states

Important:
- These types must reflect the approved governance docs, not a generic SaaS model.
- Geography and political types must align to the approved taxonomy.
- Subscription must be treated as the final entitlement gate.

Expected result:
a shared canonical type package that all later packages can depend on.

---

### Phase C — Implement `packages/core/geography`
Create the shared geography core based on the approved geography taxonomy.

At minimum implement:
- typed geography hierarchy
- ancestry model
- parent/child relationships
- helper functions for rollups and traversal
- place type definitions
- location assignment helpers
- query helpers for aggregation use cases

Must support future use cases such as:
- all markets in an LGA
- all office holders in a state
- all vendors across the country
- all motor parks in a senatorial zone

Also prepare the D1 seed-data structure direction for:
- 36 states
- 774 LGAs
- 8,814 wards
as referenced in Milestone 1 notes.

Expected result:
a reusable geography core package that can later power discovery, search, listings, and reporting.

---

### Phase D — Implement `packages/core/politics`
Create the shared political core based on the approved political taxonomy.

At minimum implement:
- political office types
- territory types
- jurisdiction model
- political assignment model
- party affiliation model interfaces
- term model
- candidate model scaffold if needed
- helper validation rules linking office types to territory types

Examples that must be representable:
- Councilor → Ward
- Local Government Chairman → LGA
- House of Assembly Member → State Constituency
- House of Representatives Member → Federal Constituency
- Senator → Senatorial District / Zone
- Governor → State
- President → Country

Expected result:
a reusable politics package that cleanly overlays geography and individuals/organizations.

---

### Phase E — Implement `packages/auth`
Create the first auth/tenancy package scaffold aligned with TDR-0008.

At minimum implement:
- JWT validation structure
- workspace-scoped access context
- tenant context model
- membership and role interfaces
- authorization guard/middleware scaffold
- entitlement-aware access evaluation scaffold
- internal service authentication placeholder structure
- secure interface boundaries for future expansion

Important:
- Do not build a sloppy global auth layer.
- Access must be explicitly tenant/workspace-aware.
- Roles alone are not enough; design for entitlement checks too.

Expected result:
a usable auth foundation package ready for future API integration.

## IMPLEMENTATION STANDARDS

### Type system
- Be explicit.
- Prefer clear interfaces and discriminated unions where useful.
- Avoid premature overengineering.

### Geography
- Model ancestry in a way that supports hierarchical queries and aggregation.
- Keep room for both administrative and facility places.

### Politics
- Model office assignment as structured data, not loose string labels.

### Auth
- Design for identity + tenant scope + entitlement, not just login.

### Database
- Keep migration files clean and reversible.
- Leave room for staging/production D1 environment separation already approved.

### Testing
Add at least foundational tests for:
- geography ancestry helpers,
- office-to-territory validation rules,
- basic entitlement-aware auth evaluation scaffolds,
- canonical type integrity where useful.
