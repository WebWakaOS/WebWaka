# Migrating a `verticals-*` Package to Engine-Only Config

> **Wave 3 — B4-2** | WebWaka OS  
> This guide explains how to retire a standalone `packages/verticals-<slug>` package and replace it with a pure engine config entry in `packages/vertical-engine/src/registry.ts`.

---

## When to Migrate

Migrate a vertical when:

- The standalone package has **no custom FSM logic** that diverges from the standard `seeded → claimed → [verified] → active → suspended` pattern.
- The package has **no extra sub-entities** beyond what the engine supports generically.
- The package's route handlers duplicate what `generateRoutes()` already produces.

If the vertical has deeply custom business logic (e.g. `verticals-bank` with multi-step KYC), keep the standalone package and use the engine for the shared parts.

---

## Step-by-Step Migration

### 1. Audit the existing package

```bash
ls packages/verticals-<slug>/src/
```

Check for:
- Custom FSM states/transitions not in the standard set
- Extra sub-entity tables not modelled in `VerticalConfig.subEntities`
- Custom middleware or guards (e.g. licence number validation)
- Non-standard profile fields (array types, JSON blobs)

### 2. Map to VerticalConfig

Open `packages/vertical-engine/src/registry.ts` and find a similar existing entry as a template. Then create your config:

```typescript
'<slug>': {
  slug: '<slug>',
  displayName: 'Human Readable Name',
  primaryPillar: 1,          // 1=Commerce, 2=Civic, 3=Government
  milestone: 'M9',
  maturity: 'full',          // 'full' | 'basic' | 'stub'
  tableName: '<slug>_profiles',
  entityType: 'organization',
  profileFields: [
    { column: 'business_name', property: 'businessName', type: 'string', required: true, label: 'Business Name' },
    // ... add all fields from the old package's DB schema
  ],
  createFields: ['businessName'],
  updateFields: ['businessName'],
  fsm: {
    states: ['seeded', 'claimed', 'active', 'suspended'],
    initialState: 'seeded',
    transitions: [
      { from: 'seeded', to: 'claimed' },
      { from: 'claimed', to: 'active' },
      { from: 'active', to: 'suspended' },
      { from: 'suspended', to: 'active' },
    ],
  },
  subEntities: [],  // add sub-entities if needed
  ai: {
    autonomyLevel: 1,
    allowedCapabilities: ['bio_generator'],
    useCases: ['Generate profile bio'],
  },
  route: { basePath: '/<slug>', entitlementLayer: 'Commerce' },
  compliance: { kycTierForClaim: 1, ndprLevel: 'standard' },
},
```

### 3. Run the parity checker

```bash
pnpm vitest run packages/vertical-engine/src/testing/parity.test.ts
```

This verifies that the engine-generated routes produce the same HTTP responses as the standalone package for the core operations (create, get, update, FSM transition).

### 4. Update the DB migration (if needed)

If the old package created its own table with extra columns, add a migration to align the schema with what the engine expects:

```sql
-- apps/api/migrations/XXXX_align_<slug>_to_engine.sql
ALTER TABLE <slug>_profiles ADD COLUMN IF NOT EXISTS status TEXT DEFAULT 'seeded';
```

### 5. Remove the standalone package routes

In `apps/api/src/route-groups/`, remove the import and `app.route()` call for the old package. `generateAllRoutes()` will handle the vertical automatically.

### 6. Delete the old package (after verification)

```bash
rm -rf packages/verticals-<slug>
# Remove from pnpm-workspace.yaml if explicitly listed
```

Then run:

```bash
pnpm install
pnpm build
pnpm test
```

### 7. Update the registry count comment

Keep the comment in `registry.ts` up to date:

```typescript
// Registry: N verticals across 3 pillars
```

## Parity Verification Checklist

Before deleting the old package, verify:

- [ ] `POST /<slug>` creates a profile and returns `201`
- [ ] `GET /<slug>/:id` returns the correct profile
- [ ] `PATCH /<slug>/:id` updates allowed fields only
- [ ] `POST /<slug>/:id/advance` triggers FSM transitions correctly
- [ ] All guards (KYC tier, licence checks) behave identically
- [ ] Sub-entities (if any) CRUD correctly
- [ ] Compliance fields (`kycTierForClaim`, `requiredLicences`) are enforced

## Known Divergences

Some packages have custom logic that the engine cannot yet replace:

| Package | Divergence | Resolution |
|---------|-----------|-----------|
| `verticals-bank` | Multi-step KYC with face match | Keep standalone; engine handles standard fields |
| `verticals-hospital` | Multi-ward sub-entity graph | Keep standalone until engine adds nested subEntities |
| `verticals-ngo` | SCUML registration workflow | Keep standalone |
