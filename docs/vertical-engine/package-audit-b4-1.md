## Audit Methodology

For each package, we compared:
1. **FSM** — states and transitions in the package vs. registry
2. **Sub-entities** — custom sub-tables not in engine config
3. **Profile fields** — extra columns not in engine's profileFields
4. **Route handlers** — custom business logic not replicated by `generateRoutes()`

## Key Findings

### Pattern 1 — Clean migration (12/20)
Most verticals (`restaurant`, `pharmacy`, `hotel`, `school`, `farm`, `church`, `mosque`, `market`, `supermarket`, `spa`, `bakery`, `car-wash`) have standard FSMs and no complex business logic. They are engine-ready after adding sub-entity configs.

### Pattern 2 — Licence-state FSM extension (4/20)
Verticals with a single extra verification state (`transport-company`, `cooperative`, `electricity-provider`, `insurance-broker`) can be migrated with a custom FSM guard. The engine supports guards via `fsm.guards[]`.

### Pattern 3 — Multi-step regulatory workflows (3/20)
`hospital`, `bank`, `ngo` have complex multi-step regulatory workflows (CBN, NHIA, SCUML) that go beyond what the engine's single-guard system can express. These should remain as standalone packages.

### Pattern 4 — Stubs pending full implementation (1/20)
`polytechnic` is a stub — migrate after full NUC accreditation workflow is designed.
